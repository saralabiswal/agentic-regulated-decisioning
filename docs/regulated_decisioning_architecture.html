Reference Architecture

Agentic Decisioning Fabric

Domain-independent regulated decision platform

Author: Sarala Biswal  ·  Repository: agentic-regulated-decisioning

A production-grade agentic decisioning platform with a constant L0–L9 architecture and pluggable domain behavior. Business users run typed Playbooks against domain adapters. The platform validates, assembles context, executes specialist agents, applies jurisdiction-aware governance, routes human review when required, and writes append-only audit records. The same runtime serves insurance, lending, healthcare, and wealth decisioning without domain logic in platform code.

Business View

One platform for four regulated domains

  • Insurance, lending, healthcare, and wealth share the same L0–L9 execution path.
  • Domain adapters own case types, agent sequence, governance YAML, and MCP configuration.
  • Playbooks give business users typed YAML control over rule overrides and test submissions without touching platform code.
  • Human review workbench provides filtered queues, assembled evidence, and append-only decision recording.

Technical View

Typed boundaries and local-first runtime

  • Pydantic models cross every layer boundary — intake, stream, graph, agent, context, governance, workbench, audit, and Playbook.
  • APP_MODE=local_sync runs the full decision path without external service dependencies.
  • APP_MODE=real activates Redis Streams and PostgreSQL when configured.
  • Real MCP connectors are intentionally config-gated — mock and fixture paths are the safe defaults.

Operations View

Observable, inspectable, and locally runnable

  • Health checks cover API, data store, stream, and MLflow availability.
  • Prometheus metrics, OpenTelemetry trace spans, and LLM token cost tracking are implemented.
  • Settings page exposes runtime mode, provider config, service links, and stream inspection.
  • MLflow model registry with deterministic fallback scoring when no model is promoted.

L0–L9 Layer Map

Implemented layers, code ownership, and behavior

Each layer reflects the files and runtime behavior currently implemented in the repository.

Implemented architecture
L0

Submission intake

api/routers/intake.py core/schemas.py core/playbook_schema.py

Incoming API submissions and Playbook uploads are normalized into typed SubmissionEvent records. Playbook YAML is parsed into a Playbook model and validated before execution begins.

  • UUID validation is enforced on all submission identifiers.
  • Domain and case type are constrained by schema and adapter validation rules.
  • Playbook validation returns blocking errors and non-blocking warnings separately.
Primary output

SubmissionEvent or structured Playbook validation response with field-level messages.

L1

Event stream

platform/stream/producer.py platform/stream/consumer.py platform/stream/inspector.py

Three practical stream modes: direct local sync execution, in-process memory stream, and Redis Streams for live service mode. Consumer failures retry three times then write a dead-letter queue record.

  • local_sync executes the handler immediately — no service dependency.
  • real activates Redis Streams when REDIS_URL is configured.
  • Stream inspection endpoint exposes inputs, pending count, DLQ count, and routing notes.
Implementation note

Kafka is available as an optional compose topology. The active backend code path is local sync, memory, or Redis Streams.

L2

Domain orchestrator

platform/orchestrator/graph.py platform/orchestrator/nodes.py domains/registry.py

A typed LangGraph state graph carries execution across context assembly, up to three named domain agents, governance evaluation, optional human review routing, and audit writing. State is maintained in OrchestratorState.

  • Routing decisions are based on context confidence, agent confidence thresholds, case value thresholds, mandatory review case types, and governance escalation overrides.
  • Domain agent sequence is supplied by the adapter — no routing logic lives in platform code.
Primary output

Final OrchestratorState carrying decision, confidence, routing path, governance result, and audit record identifier.

L3

Specialist agents

domains/*/agents.py platform/llm/client.py core/schemas.py

Agents are owned by each domain module and return typed AgentOutput with required explanation, confidence, evidence references, and flags. LLM rationale rewriting is optional — deterministic fallback applies when a provider is unavailable.

  • Insurance: triage, risk scoring, appetite check.
  • Lending: eligibility, credit scoring, policy check.
  • Healthcare: clinical triage, criteria check, coverage rules.
  • Wealth: suitability, risk tolerance, product eligibility.
LLM behavior

Provider config supports mock, auto, ollama, hosted providers with keys, or custom base URL. Decisions remain deterministic regardless of provider availability.

L4

Context and MCP abstraction

platform/mcp/assembler.py platform/mcp/servers/*.py platform/mcp/fixtures.py

The context assembler gathers core, history, and external data from three abstract MCP servers into one UnifiedContext using asyncio.gather with partial-failure tolerance. Mock and fixture paths are the default; real connectors raise until credentials and implementations are supplied.

  • Context confidence is FULL, PARTIAL, or MINIMAL based on source availability.
  • Minimal context triggers escalation before agent execution.
  • Playbook submission payload is used as the core context source in Playbook runs.
Connector status

Real connector support is a configuration contract. Vendor-specific implementations are not embedded — they are supplied at deployment time.

L5

Human review workbench

platform/workbench/queue.py api/routers/workbench.py ui/src/App.tsx

Cases requiring review are enqueued as typed WorkbenchCase objects containing submission, assembled context, agent outputs, recommendation, and escalation reason. Reviewers see a prepared workspace — no context reconstruction is required.

  • Reviewer actions: confirm recommendation, override with notes, request more information.
  • Human decisions append new audit records — prior automated records are never modified.
  • Cases persist in SQLite locally or PostgreSQL when configured.
Primary output

Reviewer action record plus an appended AuditRecord with decision_type: human_override or human_confirm.

L6

Data platform

platform/data/store.py platform/data/postgres_store.py platform/data/migrations.py

SQLite at .local/regulated_decisioning.db is the default local path. PostgreSQL activates when DATABASE_URL starts with postgresql. Tables cover audit records, workbench cases, DLQ records, LLM costs, model versions, and Playbook run history.

  • SQLite keeps the repository runnable without any external service dependencies.
  • PostgreSQL helpers use async connections for live service mode.
  • Audit inserts are append-only by design — no update or delete paths exist.
Default path

sqlite:///.local/regulated_decisioning.db

L7

Model registry and local ML runtime

platform/registry/model_registry.py platform/registry/model_runtime.py scripts/train_local_model.py

MLflow-backed model metadata and scoring with deterministic rules fallback when no model is promoted. Registration, promotion, and rollback events write append-only model audit records. Training supports sklearn, gradient boosting, pyfunc, and optional open-source runtimes.

  • No model is required for the application to run — rules fallback is always available.
  • Shadow mode runs a staging model alongside production and records comparison metrics.
  • Scoring attempts native probability output first, then generic MLflow pyfunc.
Fallback behavior

Deterministic rules scoring activates automatically when no promoted MLflow model is found for a domain.

L8

Observability

platform/observability/*.py api/routers/health.py grafana/dashboards/*.json

Health checks, Prometheus metrics, OpenTelemetry trace spans, and LLM token cost tracking. Health endpoint reports database, Redis, and MLflow availability. Metrics export at /metrics. Jaeger OTLP configured through JAEGER_ENDPOINT.

  • Trace spans wrap FastAPI, MCP source calls, agent execution, governance, audit, and Playbook execution.
  • LLM token cost rows are persisted when provider usage is returned.
  • Prometheus and Grafana dashboards are provided by the Docker compose stack.
UI page

Operations page surfaces health posture, model registry stage, and last run details with links to Grafana and Jaeger.

L9

Governance and audit

platform/governance/engine.py platform/governance/audit_trail.py domains/*/governance/*.yaml

Loads jurisdiction-aware YAML rules, checks prohibited evidence fields, enforces explanation presence, evaluates escalation override conditions using a safe expression evaluator, and records the applied rule set. Audit records are append-only for automated decisions, human actions, and model lifecycle events.

  • Safe expression evaluator supports comparisons, boolean logic, and containment checks — no eval().
  • Playbooks merge additional prohibited factors and mandatory review triggers on top of jurisdiction defaults.
  • Audit and Reports page reconstructs the full decision package from persisted records and layer events.
Primary output

AuditRecord plus report and trace package reconstructible from persisted records alone.

Domain Plugins

Same platform, domain-owned behavior

All domain-specific logic lives in domains/. The platform queries each adapter for supported case types, agent sequence, escalation thresholds, governance rules, MCP config, demo cases, and Playbook validation.

Architectural claim: No platform code in platform/, api/, or core/ contains domain-specific logic. Domain is a plugin. The platform is constant.

Proof: pytest tests/test_domain_protocol.py -v — asserts all four adapters satisfy the DomainAdapter protocol and that no platform module imports from a domain-specific module.

Playbook Execution

Business-owned YAML, platform-owned runtime

The Playbook flow is the primary product path. A business user defines domain, jurisdiction, rule overrides, and a test submission. The platform owns all technical execution.

Templates in static/playbook_templates/
StepWhat happensCode pathOutput
1. Select or upload User selects a domain-specific template or uploads a YAML Playbook file. ui/src/App.tsx · api/routers/playbook.py Raw YAML content
2. Validate YAML is parsed, schema constraints are enforced, and the domain adapter validates required submission fields. Blocking errors and non-blocking warnings are returned separately. core/playbook_schema.py · domains/*/adapter.py Structured ValidationMessage list
3. Merge overrides Playbook thresholds and mandatory review triggers are merged with domain adapter defaults. Playbook rules can tighten but not loosen jurisdiction-level prohibitions. platform/playbook/executor.py PlaybookOverride
4. Execute graph The orchestrator loads the domain adapter, assembles context, runs specialist agents, applies governance gates, and routes human review when required. platform/orchestrator/graph.py Final OrchestratorState
5. Persist and trace Layer events, governance rule events, audit records, workbench cases, and run history are written. SSE stream delivers real-time layer updates to the UI. platform/playbook/events.py · platform/governance/audit_trail.py Result · Report · Audit package · Trace

Runtime Modes

Three operational configurations

The default path runs without any external service dependencies. Live services are opt-in through environment configuration.
